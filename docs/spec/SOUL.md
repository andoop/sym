# Sym 之魂（Soul）

本文档概括 **Sym** 的追求与核心目标，供设计与评审时对照；实现细节仍以代码与 `README.md` 为准。

---

## 我们为什么做 Sym

- **人机协同**：语言要写得清、查得动、模型和工具容易跟着推理，而不是堆满隐式规则和「只能意会」的惯用法。
- **小但完整**：具备词法、语法、类型、可运行语义，能支撑真实示例（多文件、prelude、内建与 Sym 源码并存）。
- **歧义少**：语法与语义尽量一眼能对上，减少「看起来像 A、其实是 B」的坑。

---

## 核心原则：语法直接对应实现

**表面写法应贴近运行时的形状，拒绝不必要的隐藏分配与魔法。**

- **代数类型（ADT）**：`type` 里有什么变体、什么字段，值就是那样构造、那样在 `match` 里拆开。
- **列表即链表**：`List[T]` 由 `Nil` 与 `Cons(head, tail)` 显式构成；手写目录就是嵌套 `Cons`，而不是偷偷变成数组或向量。
- **内建保持中性**：宿主提供通用能力（如 I/O、字符串、HTTP、环境变量等），**具体产品逻辑（例如某家 API 的 URL 与 JSON 形状）写在 Sym 源码里**，而不是写进编译器/解释器当「专用魔法」。
- **可有糖，但不改语义**：若将来增加字面量或语法糖，应**脱糖**到同一套显式语义上，而不是引入第二套运行时模型。

我们相信：**可预测的行为**比**少敲几个字符**更配得上「给 AI 与工具用的语言」这一定位。

---

## 与原则一致的设计取舍

- **绑定而非可变变量**：`let` 与参数是名字到值的绑定，避免隐式可变状态干扰推理。
- **显式控制流**：`if`/`match`/`while`/`do` 边界清楚；短路逻辑在语义上可说明白。
- **类型与检查**：在小型语言范围内做有意义的静态检查（含 ADT 与 `match` 穷尽性等），让「能过检查」本身成为一种文档。

---

## 成功时 Sym 像什么

- 读一段 Sym，能大致说出**内存里长什么样**、**下一步执行什么**。
- 工具和模型能稳定地做**搜索、重构、生成与审阅**，少碰「只有老用户才知道」的角落。
- 示例（如图书管理、外部 HTTP 客户端）**主要逻辑在 `.sym` 里**，宿主只提供**薄而通用**的原语。

---

## 刻意不追求什么（当前阶段）

- 不等于「语法最省」或「一行写完」的脚本语言。
- 不追求隐式全局可变、不透明运行时或「帮你智能优化到认不出来」的魔法。
- 工业级后端（JIT/AOT）、庞大标准库、完整 IDE 协议等可作为**后续路线**，但不稀释上述灵魂条款。
